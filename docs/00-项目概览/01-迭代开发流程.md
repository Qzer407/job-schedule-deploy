# 迭代开发流程

## 1. 流程概述

每次迭代开发都需要遵循统一的流程，确保项目可追溯、可管理。

### 1.1 迭代开发流程图

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  需求确认   │───►│   设计文档  │───►│   开发实现  │───►│   代码检查  │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
       │                                              │
       │                                              ▼
       │              ┌─────────────┐    ┌─────────────┐
       │              │  Git推送   │───►│  文档更新   │
       │              └─────────────┘    └─────────────┘
       │                    │
       ▼                    ▼
┌─────────────┐    ┌─────────────┐
│  发布报告   │◄───│  迭代总结   │
└─────────────┘    └─────────────┘
```

## 2. 迭代前准备

### 2.1 需求确认

在迭代开始前，需要确认以下内容：

| 确认项 | 说明 | 检查清单 |
|--------|------|----------|
| 功能需求 | 本次迭代需要完成的功能点 | [ ] 功能列表已明确 |
| 优先级 | 功能优先级排序 | [ ] 优先级已确定 |
| 依赖关系 | 功能之间的依赖 | [ ] 依赖关系已梳理 |
| 资源评估 | 所需人力和时间 | [ ] 评估已完成 |

### 2.2 创建迭代分支

```bash
# 基于 main 分支创建迭代分支
git checkout -b feature/v0.x.0

# 或者从当前分支创建
git checkout -b feature/v0.4.0-task-orchestration
```

## 3. 设计阶段

### 3.1 设计文档输出

每次迭代开发前，需要输出以下设计文档：

| 文档 | 说明 | 模板位置 |
|------|------|----------|
| 迭代需求文档 | 本次迭代的需求清单 | `04-落地实现/` |
| 接口设计文档 | API 接口设计 | `02-接口与规范/` |
| 模块设计文档 | 核心模块设计 | `模块设计/` |

### 3.2 设计文档模板

迭代设计文档应包含：

```markdown

- 迭代周期：YYYY-MM-DD ~ YYYY-MM-DD
- 迭代负责人：[姓名]

## 4. 开发阶段

### 4.1 开发规范

1. **代码规范**：遵循项目编码规范文档
2. **提交规范**：使用 Conventional Commits
3. **测试规范**：核心功能需要编写单元测试

### 4.2 Git 提交规范

```bash
# 功能开发
git commit -m "feat: 新增任务管理功能"

# 修复bug
git commit -m "fix: 修复任务列表分页问题"

# 文档更新
git commit -m "docs: 更新API接口文档"

# 代码重构
git commit -m "refactor: 重构任务服务代码"

# 性能优化
git commit -m "perf: 优化数据库查询性能"
```

### 4.3 开发检查清单

| 检查项 | 说明 | 状态 |
|--------|------|------|
| 代码规范 | 遵循编码规范 | [ ] |
| 单元测试 | 核心功能有测试 | [ ] |
| API 文档 | Swagger 文档完整 | [ ] |
| 错误处理 | 异常情况已处理 | [ ] |

## 5. 代码检查阶段

### 5.1 检查范围

每次迭代完成后，需要检查以下内容：

| 检查类别 | 检查项 | 检查清单 |
|---------|--------|----------|
| **代码完整性** | 后端文件完整 | [ ] Controller/Service/Entity 完整 |
| | 前端文件完整 | [ ] 组件/API 完整 |
| | 数据库脚本完整 | [ ] schema.sql 更新 |
| **代码一致性** | 文档与代码一致 | [ ] 接口路径、字段名称一致 |
| | 前后端接口对应 | [ ] API 路径一致 |
| **代码质量** | 代码规范 | [ ] 命名、注释符合规范 |
| | 无冗余代码 | [ ] 无废弃代码 |
| **功能验证** | 功能正常运行 | [ ] 基本功能测试通过 |

## 6. 迭代完成阶段

### 6.1 Git 推送

```bash
# 提交所有变更
git add .
git commit -m "feat: v0.x.0 迭代完成"

# 推送到远程仓库
git push origin feature/v0.x.0

# 合并到主分支
git checkout main
git merge feature/v0.x.0
git push origin main
```

### 6.2 文档更新

| 文档 | 更新内容 | 更新位置 |
|------|---------|---------|
| 迭代计划 | 标记迭代状态 | `00-项目概览/00-迭代计划.md` |
| 模块设计 | 新增/更新模块设计 | `模块设计/` |
| API接口设计 | 新增接口 | `02-接口与规范/01-API接口设计.md` |
| 数据库设计 | 新增/修改表 | `01-需求与设计/04-数据库设计.md` |

---

## 8. 注意事项

1. **每次迭代都必须输出检查报告**，确保项目可追溯
2. **文档与代码必须保持一致**，发现问题及时修复
3. **Git 提交信息必须规范**，方便后续追溯
4. **遗留问题必须记录**，避免遗漏
